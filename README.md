# Сортировщик лент

Внешняя сортировка последовательности 32-битных целых чисел с ограниченной RAM: вход и выход - бинарные файлы подряд идущих `std::int32_t`, промежуточные данные - «ленты» через общий интерфейс с файловой реализацией.

Требования: **CMake ≥ 3.30**, компилятор с **C++20**.

---

## Сборка

Из корня репозитория:

```bash
cmake -S . -B build
cmake --build build
```

Опционально выключить тесты
```bash
cmake -S . -B build -DBUILD_TESTING=OFF
```

---

## Запуск

Формат аргументов:

```text
<program> <input.bin> <output.bin> [config.ini]
```

- **`input.bin`** - размер кратен `sizeof(std::int32_t)`, иначе приложение сообщит об ошибке при открытии ленты.
- **`output.bin`** создаётся/перезаписывается с той же длиной в ячейках, что у входа.
- **`config.ini`** необязателен. Если третий аргумент не передан, при **наличии** в текущей рабочей директории файла **`tape_config.ini`** он загружается автоматически, если файла нет - используются значения по умолчанию из `tatlin::AppConfig`.

Рабочая директория важна: по умолчанию конфиг ищется относительно **текущего каталога** процесса, временные ленты пишутся в **`tmp`** под тем же текущим путём (см. `main.cpp`).

### Конфиг (`key=value`, строки с `#` - комментарии)

Поддерживаемые ключи (см. `load_config` в `src/app_config.cpp`):

| Ключ | Смысл |
|------|--------|
| `read_delay_ms`, `write_delay_ms`, `rewind_delay_ms`, `shift_delay_ms` | Задержки операций над файловой лентой, целые неотрицательные, мс |
| `memory_limit_bytes` | Верхняя граница памяти под чанки сортировки (байты), должно давать хотя бы 2 ячейки `int32` |
| `temporary_tapes_k` | Максимальное число временных лент|

## Архитектура и зависимости

**`TapeSorter`** работает только с абстрактной лентой **`Tape`** и фабрикой  **`TapeFactory`**: читает вход, пишет в переданную выходную ленту, заводит временные ленты через фабрику. О каком носителе это («файл на диске», пути, симуляция задержек через **`AppConfig`**) объект сортировщика по смыслу не знает - такую реализацию ему подставляют **`FileTape`** и **`FileTapeFactory`**, которые создаются и настраиваются в **`main`**.

Иначе говоря, алгоритм опирается на интерфейсы, а не на конкретный тип ленты: для тестов или другой подсистемы хватит своей пары классов **`Tape`** + **`TapeFactory`**, без переписывания **`tape_sorter.cpp`**.


## Алгоритм сортировки

Адаптированный под условия и ограничения (количество хранимых в RAM интов и количество создаваемых временных тейпов) задачи k-way merge-sort

### Параметры

Выбирается чётное число временных лент:

`tape_count = min(configured temporary tapes k, in-memory chunk capacity in int32 cells, 128)`

затем округление до чётного числа не меньше 4 (иначе выбрасывается ошибка конфигурации - для схемы с двумя банками по `tape_count / 2` лент недостаточно менее четырёх носителей).

«В памяти помещается» столько элементов, сколько задаёт **`memory_limit_bytes / sizeof(std::int32_t)`** - именно этого размеров чанки сортируются в RAM (`std::ranges::sort`).

### Фаза 1: Формирование отрезков (runs)

С входной ленты последовательно читаются чанки (размера ранее упомянутого). Каждый чанк сортируется и записывается на следующую временную ленту активного банка (первую половину временных очередей) в режиме кругового обхода (циклически). Для каждой временной ленты хранится список длин записанных на неё подряд отрезков.

### Фаза 2: Слияние через переливание во второй банк

Мерджим первые отрезки (runs) первой половины обычным k-way merge-sort.
Полученные таким образом отрезки циклически кладем в ленты второй половины. Как только мы слили все отрезки активного банка, меняем банки местами (помечаем активным тот, что был не активен и наоборот). Продолжаем, пока не останется один единственный run в каком-то временном тейпе какого-то из банков.

### Фаза 3: Вывод

Когда во всём активном банке один непустой отрезок, он копируется на выходную ленту.
